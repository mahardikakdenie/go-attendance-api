# Antigravity Agent Rules & Project Context

Selamat datang di proyek **go-attendance-api** (B2B SaaS Multi-Tenant Attendance and Payroll System). Harap ikuti aturan dan panduan di bawah ini saat memberikan solusi, membuat kode, atau meninjau arsitektur dalam workspace ini.

---

## 1. Domain Knowledge (SaaS Multi-Tenant)

- **Arsitektur Multi-Tenant:**
  - Sebagian besar tabel transaksi memiliki `tenant_id`. Setiap query data transaksi **harus** difilter berdasarkan `tenant_id` user yang sedang aktif untuk menjamin isolasi data (data leak prevention).
  - **Tenant ID 1** adalah **SaaS Headquarters** (Tenant khusus platform/Superadmin). Hanya user di Tenant ID 1 yang dapat mengelola modul platform global (Trial, Provisioning, Global Billing/Subscription, dll.).

- **Hierarki Role Utama (Base Roles):**
  - `SUPERADMIN`: Hak akses penuh ke seluruh platform (Tenant 1) dan tenant-specific administration.
  - `ADMIN`: Hak administrator level tenant.
  - `HR`: Pengelolaan data karyawan, kehadiran, dan departemen level tenant.
  - `FINANCE`: Pengelolaan payroll, tunjangan, dan klaim biaya (expenses) level tenant.
  - `EMPLOYEE`: Akses dasar (misal check-in/out, melihat payslip sendiri, request cuti).

---

## 2. Panduan Backend (Golang)

- **Standardisasi Response API:**
  Semua endpoint API harus mengembalikan format payload JSON yang seragam:
  ```json
  {
    "success": true,
    "meta": {
      "message": "Success",
      "code": 200,
      "status": "OK",
      "pagination": {
        "total": 12,
        "per_page": 10,
        "current_page": 1
      }
    },
    "data": []
  }
  ```
  *(Catatan: Field `pagination` hanya disertakan jika endpoint mendukung list/query pagination).*

- **Pagination & Filtering:**
  - Terima parameter `limit`, `offset` (atau `page`), dan `search` (pencarian case-insensitive & partial match pada kolom relevan).
  - Pastikan nilai `pagination.total` mencerminkan jumlah baris yang cocok setelah filter/search diterapkan, bukan total baris tabel tanpa filter.

- **Keamanan (SQL Injection & Validasi):**
  - Gunakan parameterized query (atau ORM safely) untuk mencegah SQL injection.
  - Selalu lakukan validasi kepemilikan tenant (`tenant_id`) untuk endpoint non-superadmin.

---

## 3. Panduan Frontend (TypeScript & UI/UX)

- **Integrasi API:**
  - Gunakan wrapper request terenkripsi/aman (misal `secureRequest` di folder `src/service/`).
  - Kirim parameter filter yang bersih (abaikan filter kosong/default `"all"` sebelum dikirim ke API).

- **Permission-Based UI Rendering:**
  - Tombol aksi (tambah/edit/hapus/akses menu) wajib dikontrol secara dinamis menggunakan daftar permission user (diperoleh dari `/me` endpoint, misal `user.create`, `user.edit`, `user.delete`).

- **UX Standards:**
  - Sediakan visualisasi Empty State jika data kosong.
  - Selalu tangani error umum seperti `409 Conflict` (email/ID duplikat) atau `403 Forbidden` dengan pesan error yang ramah pengguna.

---

## 4. Alur Kerja Agent (Golden Rules)

1. **Gunakan Konteks Template:** Jika mengerjakan task baru, periksa apakah ada rancangan detail di `.github/ISSUE_TEMPLATE` yang membahas modul tersebut.
2. **Jangan Merusak Struktur Data:** Hindari breaking changes pada struktur response JSON yang sudah disepakati oleh Frontend dan Backend.
3. **Pemberian Solusi Bertahap:** Selalu buat perencanaan perubahan skema database (migrasi) terlebih dahulu sebelum mengimplementasikan logika endpoint atau antarmuka baru.
